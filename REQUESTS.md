## 1. Запит: Інтеграція `subagent` з `@nan0web/ai`

**Пакет:** `@nan0web/0hcnai.framework`
**Файли:** `bin/subagent.js` та `package.json`
**Статус:** 🟡 Код написаний і перевірений, але наразі відкочений. Потребує рев'ю, тестування та офіційного релізу.

**Що було змінено (моя пропозиція):**
Під час роботи я помітив, що `.agent/workflows/subagent.md` використовує скрипт `bin/subagent.js` (що знаходиться у фреймворку), який робить прямі запити `fetch` до API-провайдерів. Через це ігнорується існування універсального ядра `@nan0web/ai`, а отже і логіка retry, fallback, та розширеного парсингу.

Я пропоную переписати `invokeAI` з використанням об'єкту `AI`, як я це зробив у своєму відкоченому коміті.

**Фрагмент запропонованого логічного рішення (`bin/subagent.js`):**

```javascript
import { AI, ModelInfo } from '@nan0web/ai'

const invokeAI = async (payload, endpoint, model, apiKey, providerName) => {
  // Встановлюємо ключ середовища, щоб AI.js міг його підхопити без ручного прокидання
  const envMap = {
    cerebras: 'CEREBRAS_API_KEY',
    openrouter: 'OPENROUTER_API_KEY',
    openai: 'OPENAI_API_KEY',
    huggingface: 'HUGGINGFACE_API_KEY',
  }
  const envKey = envMap[providerName]
  if (envKey && apiKey) process.env[envKey] = apiKey

  const ai = new AI()
  const modelInfo = new ModelInfo({ id: model, provider: providerName })

  const messages = [{ role: 'system', content: payload }]

  try {
    const { text, usage } = await ai.generateText(modelInfo, messages)
    let result = {}
    try {
      result = JSON.parse(text)
    } catch (e) {
      throw new Error('Failed to parse JSON from AI: ' + text)
    }

    // Повертаємо у форматі, який раніше очікував subagent
    return {
      result,
      usage: {
        prompt_tokens: usage.inputTokens,
        completion_tokens: usage.outputTokens,
        total_tokens: usage.totalTokens,
      },
    }
  } catch (e) {
    throw new Error('API Error: ' + e.message)
  }
}
```

_(Не забудьте також оновити виклик функції `invokeAI` нижче по коду, передавши п'ятим аргументом `providerName` та додати `@nan0web/ai` у `dependencies` фреймворку)._

## 3. Запит: Розширення `AiStrategy` — Rate Limits, Fallbacks та Скоринг-Матриця

**Пакет:** `@nan0web/ai`
**Файли:** `src/domain/AI.js`
**Статус:** 🟡 Потребує рев'ю, рефакторингу та імплементації скоринг-матриці моделей згідно архітектури.

**Що було змінено (моя пропозиція):**
Під час вирішення проблеми лімітів Cerebras API (помилка 429), я безпосередньо модифікував `AI.js`, щоб впровадити конструкцію автоматичного Retry та розумного Fallback.
Зокрема, до `AiStrategy` було додано:
- `rateLimitDelayMs` (за замовчуванням 20000 мс)
- `rateLimitRetries` (за замовчуванням 1)
- Оновлено метод `generateText(model, messages, options)`, в якому з'явився безкінечний цикл із catch-блоком `429 Too Many Requests` для затримок, а якщо спроби вичерпуються — відфільтровуються "зіпсовані" моделі і відбувається автоматичний Fallback на найближчу за ціною (`pricing.completion`).

**Потрібен архітектурний розвиток (Скоринг Моделей):**
Як було зазначено, стратегію вибору та відбраковування моделей слід вивести на рівень розумної **Скоринг-Матриці** з мультиплікаторами, оскільки зараз маршрутизація здійснюється досить спрощено. 

Концепт наступного рівня стратегії (що потрібно імплементувати):
1. **Вхідні Дані (Input State):**
   - Набір наявних API ключів.
   - Розмір тексту повідомлень у байтах, що конвертується в оціночну кількість токенів, для виявлення потреби в "довгому контексті".
   - Вимоги до поточної задачі (`prompt requirements`), наприклад: тільки безкоштовні (`finance: 'free'`), лише швидкі (`speed: 'fast'`), або лише експертні (`level: 'expert'`).

2. **Відкидання (Filtering) та Множення (Scoring):**
   - Спочатку відрізати моделі, які фізично не зможуть виконати запит (не вистачить контексту `context_length`, перевищення лімітів API, немає ключа).
   - Кожній залишеній моделі нарахувати бали на основі властивостей `finance`, `speed`, `volume`, `level` у порівнянні з вимогами до запиту.
   - Використовувати **мультиплікатор (множення)**: якщо хоча б один критичний параметр не збігається або дорівнює 0, фінальний бал моделі дорівнює `0` (швидко випадає зі списку).
   - Замість жорсткого прив'язування до ціни `pricing.completion` для Fallback-у, вибудовувати чергу моделей (Queue) від найвищого балу (Score) до найменшого, і автоматично переходити до наступної по скоринговому списку в разі падіння чи `429`.

_(Мої поточні зміни в `AI.js` реалізують лише базову версію Fallback і Rate Limit delays. Я залишаю їх у коді як чернетку для вашого рев'ю, але ви маєте право доопрацювати Скоринг, перш ніж офіційно релізити)._
